# 12 — 문제 해결 가이드

## 개요

이 가이드는 OXP 플랫폼의 개발, 배포, 일상 운영 중 발생하는 일반적인 문제와 각각에 대한 실용적인 해결책을 다룹니다.

---

## 1. 로그인 문제

### 1.1 로그인 불가 — 잘못된 자격증명

**증상**: 로그인 시 "잘못된 자격증명" 오류가 반환됩니다.

**해결책**:
1. 이메일 주소와 비밀번호가 정확한지 확인합니다 (대소문자 구분).
2. 데이터베이스에 사용자 계정이 존재하는지 확인합니다.
3. 계정이 `suspended` 또는 `banned` 상태인지 확인합니다 (정지/차단 시 로그인 불가).
4. 시딩된 관리자 자격증명을 사용하는 경우 `database/seeders/UserSeeder.php`에서 기본값을 확인합니다.

### 1.2 관리자가 관리자 패널에 접근할 수 없음

**증상**: 관리자가 `/admin`으로 이동하지만 로그인 페이지로 리다이렉트되거나 "Unauthorized"가 표시됩니다.

**해결책**:
1. `users` 테이블에서 사용자의 `role = admin`인지 확인합니다.
2. `.env`의 `APP_URL`이 사용 중인 URL과 일치하는지 확인합니다.
3. `config/auth.php`의 Filament 가드 설정을 확인합니다.
4. 캐시 지우기: `php artisan optimize:clear`

### 1.3 세션이 너무 빨리 만료됨

**증상**: 예상보다 빨리 로그아웃됩니다.

**해결책**:
1. `.env`의 `SESSION_LIFETIME` 확인 (기본값: 120분).
2. 세션 드라이버가 올바르게 설정되어 있는지 확인합니다.
3. 데이터베이스 세션을 사용하는 경우 `sessions` 테이블이 존재하는지 확인합니다.

### 1.4 비밀번호 재설정 이메일을 받지 못함

**증상**: 사용자가 비밀번호 재설정을 요청하지만 이메일을 받지 못합니다.

**해결책**:
1. 프로덕션에서 `MAIL_MAILER`가 `log` 또는 `null`이 아닌지 확인합니다.
2. `MAIL_FROM_ADDRESS`가 유효한 발신자 주소인지 확인합니다.
3. `storage/logs/laravel.log`에서 이메일 오류를 검토합니다.
4. **관리자 패널 → 시스템 → 이메일 설정 → 테스트 이메일 발송**으로 이메일 설정을 테스트합니다.
5. 스팸 폴더를 확인합니다.

## 2. 제품 저장 / 관리자 패널 문제

### 2.1 제품이 저장되지 않음

**증상**: 제품 양식에서 저장을 클릭하면 오류가 표시되거나 저장되지 않고 리다이렉트됩니다.

**해결책**:
1. 양식에 표시된 유효성 검사 오류를 확인합니다 (필수 필드, 형식 문제).
2. 제품 `slug`가 고유한지 확인합니다 (중복 슬러그는 유효성 검사에 실패합니다).
3. 자세한 오류 메시지를 위해 `storage/logs/laravel.log`를 확인합니다.
4. 데이터베이스 권한을 확인합니다.

### 2.2 관리자 패널에서 이미지 업로드 실패

**증상**: 제품 이미지 또는 기사 커버 이미지 업로드 시 오류가 반환됩니다.

**해결책**:
1. 선택된 스토리지 디스크(`FILESYSTEM_DISK`)가 올바르게 설정되어 있는지 확인합니다.
2. Azure의 경우: `AZURE_STORAGE_NAME`, `AZURE_STORAGE_KEY`, `AZURE_STORAGE_CONTAINER`가 올바른지 확인합니다.
3. 로컬의 경우: `php artisan storage:link`를 실행하고 심볼릭 링크가 존재하는지 확인합니다.
4. 파일 크기가 한도 내인지 확인합니다.
5. `php.ini`에서 PHP의 `upload_max_filesize` 및 `post_max_size`를 확인합니다.

### 2.3 리치 텍스트 편집기가 로드되지 않음

**증상**: Tiptap 편집기가 관리자 패널에서 빈 화면이거나 작동하지 않습니다.

**해결책**:
1. 브라우저를 강제로 새로 고침합니다 (Ctrl+Shift+R / Cmd+Shift+R).
2. JavaScript 오류를 위해 브라우저 콘솔을 확인합니다.
3. 브라우저가 최신 버전이고 JavaScript가 활성화되어 있는지 확인합니다.

## 3. 장바구니 및 결제 문제

### 3.1 장바구니에서 항목이 사라짐

**증상**: 항목을 추가한 후 장바구니가 비어 있습니다.

**해결책**:
1. 비회원의 경우: 장바구니는 브라우저 세션에 저장됩니다 — 쿠키가 활성화되어 있어야 합니다.
2. 인증된 사용자의 경우: 데이터베이스의 `carts` 및 `cart_items` 테이블을 확인합니다.
3. `SANCTUM_STATEFUL_DOMAINS`에 프론트엔드 도메인이 포함되어 있는지 확인합니다.
4. 브라우저 개발자 도구(네트워크 탭)에서 API 오류를 확인합니다.

### 3.2 주소 오류로 결제 실패

**증상**: 결제 양식에서 주소 유효성 검사 오류가 표시됩니다.

**해결책**:
1. 모든 필수 주소 필드가 입력되었는지 확인합니다.
2. NZ Post 주소 유효성 검사를 사용하는 경우 `NZPOST_ENABLED` 및 자격증명을 확인합니다.
3. 문제가 지속되면 NZ Post를 비활성화합니다: `NZPOST_ENABLED=false` 설정.

### 3.3 배송 옵션이 로드되지 않음

**증상**: 결제 시 배송 옵션이 표시되지 않습니다.

**해결책**:
1. `NZPOST_ENABLED=true`인 경우: NZ Post API 자격증명이 유효한지 확인합니다.
2. `STORE_STANDARD_SHIPPING_RATE`에 고정 요금 배송비가 설정되어 있는지 확인합니다.
3. API 엔드포인트를 확인합니다: `POST /api/store/shipping-options` 응답.

## 4. 이메일 발송 문제

### 4.1 이메일이 발송되지 않음

**증상**: 트랜잭션 이메일(주문 확인, 비밀번호 재설정)을 받지 못합니다.

**해결책**:
1. 개발 환경에서: `storage/logs/laravel.log` 확인 — `MAIL_MAILER=log`인 경우 이메일이 로그에 기록됩니다.
2. 프로덕션에서: 이메일 설정(관리자 패널) 또는 `.env`에서 SMTP 자격증명을 확인합니다.
3. **관리자 패널 → 시스템 → 이메일 설정 → 테스트 이메일 발송**으로 테스트합니다.
4. 이메일 스팸/정크 폴더를 확인합니다.
5. `MAIL_FROM_ADDRESS`가 차단되거나 위조된 도메인이 아닌지 확인합니다.

### 4.2 이메일이 발송됐지만 수신되지 않음

**증상**: 이메일 로그에는 "발송됨"이 표시되지만 수신자가 받지 못합니다.

**해결책**:
1. 스팸/정크 폴더를 확인합니다.
2. 발신 주소가 메일 공급자를 통한 발송이 승인되었는지 확인합니다 (SPF/DKIM 레코드).
3. 다른 수신자 주소로 테스트합니다.
4. 관리자 패널에서 이메일 로그를 확인합니다: `이메일 → 이메일 로그`.

## 5. 파일 업로드 문제

### 5.1 커뮤니티 게시물 첨부 파일 업로드 실패

**증상**: 커뮤니티 게시물에 파일을 첨부할 때 오류가 발생합니다.

**해결책**:
1. 파일 유형이 허용되는지 확인합니다 (`IDEA_MEDIA_ALLOWED_EXTENSIONS`).
2. 파일 크기가 한도 내인지 확인합니다 (`IDEA_MEDIA_MAX_FILE_SIZE_KB`).
3. 스토리지 디스크가 설정되고 접근 가능한지 확인합니다.
4. `ALLOW_GUEST_UPLOAD=false`가 비인증 업로드를 차단하고 있는지 확인합니다.
5. 자세한 오류를 위해 `storage/logs/laravel.log`를 검토합니다.

## 6. 번역 / i18n 문제

### 6.1 텍스트가 잘못된 언어로 표시되거나 번역 키가 표시됨

**증상**: 실제 번역 텍스트 대신 `community.posts.title`이 표시됩니다.

**해결책**:
1. 관련 `messages/{locale}.json` 파일에 번역 키가 존재하는지 확인합니다.
2. 프론트엔드 디렉토리에서 `npm run check:i18n`을 실행하여 누락된 번역을 찾습니다.
3. 번역 파일 업데이트 후 프론트엔드를 재빌드합니다: `npm run build`.

### 6.2 선택한 언어로 콘텐츠가 표시되지 않음

**증상**: 한국어나 중국어를 선택해도 제품 또는 소재에 영어 텍스트가 표시됩니다.

**해결책**:
1. 관리자 패널에서 콘텐츠의 다국어 필드가 입력되어 있는지 확인합니다.
2. 비영어 필드가 비어 있으면 시스템이 영어로 대체됩니다 — 이는 예상된 동작입니다.
3. 콘텐츠 항목의 모든 언어 필드를 입력합니다.

## 7. 캐시 문제

### 7.1 저장 후 변경 사항이 반영되지 않음

**증상**: 관리자가 설정이나 콘텐츠를 저장했지만 웹사이트에 여전히 이전 데이터가 표시됩니다.

**해결책**:
```bash
php artisan optimize:clear
php artisan config:cache
php artisan route:cache
```

### 7.2 설정 변경 후 애플리케이션 오류

**증상**: `.env` 수정 후 500 오류가 발생합니다.

**해결책**:
```bash
php artisan optimize:clear
php artisan optimize
```

## 8. 데이터베이스 마이그레이션 문제

### 8.1 테이블이 이미 존재한다는 오류로 마이그레이션 실패

**증상**: `php artisan migrate`에서 "Table already exists" 오류가 반환됩니다.

**해결책**:
1. 마이그레이션 상태 확인: `php artisan migrate:status`
2. 마이그레이션이 부분적으로 실행된 경우 `php artisan migrate:rollback`을 사용하여 마지막 배치를 실행 취소합니다.
3. 절대적으로 필요하지 않은 한 `migrations` 테이블에서 마이그레이션 레코드를 수동으로 삭제하지 마세요.

### 8.2 외래 키 오류로 마이그레이션 실패

**증상**: "Cannot add foreign key constraint" 오류로 마이그레이션이 실패합니다.

**해결책**:
1. 마이그레이션을 순서대로 실행합니다 — 외래 키 마이그레이션은 상위 테이블이 먼저 존재해야 합니다.
2. MySQL 엄격 모드가 적절하게 설정되어 있는지 확인합니다.
3. 마이그레이션 파일에서 올바른 테이블 및 컬럼 이름을 확인합니다.

## 9. 프론트엔드 빌드 문제

### 9.1 타입 오류로 빌드 실패

**증상**: `npm run build`가 TypeScript 오류로 실패합니다.

**해결책**:
```bash
# 타입 명시적 확인
npm run typecheck

# i18n 키 확인
npm run check:i18n
```
빌드를 시도하기 전에 보고된 모든 타입 오류를 수정합니다.

### 9.2 i18n 키 누락으로 빌드 실패

**증상**: 빌드 출력에 누락된 번역 키가 표시됩니다.

**해결책**:
1. `npm run check:i18n`을 실행하여 모든 누락된 키를 확인합니다.
2. 세 가지 언어 파일 모두에 누락된 키를 추가합니다 (`en.json`, `ko.json`, `zh.json`).
3. 재빌드합니다.

### 9.3 프론트엔드가 백엔드 API에 연결할 수 없음

**증상**: 모든 API 호출에서 브라우저 콘솔에 네트워크 오류 또는 CORS 오류가 반환됩니다.

**해결책**:
1. 프론트엔드 `.env`의 `NEXT_PUBLIC_API_URL`이 백엔드 URL과 일치하는지 확인합니다.
2. 백엔드 `CORS_ALLOWED_ORIGINS`에 프론트엔드 출처가 포함되어 있는지 확인합니다.
3. 백엔드가 실행 중이고 접근 가능한지 확인합니다.
4. 프론트엔드와 백엔드 간의 HTTPS/HTTP 불일치를 확인합니다.

## 10. 500 서버 오류

### 10.1 일반 500 오류

**증상**: API가 `500 Internal Server Error`를 반환합니다.

**해결책**:
1. 전체 오류 추적을 위해 `storage/logs/laravel.log`를 확인합니다.
2. 개발 환경에서 API 응답에서 자세한 오류를 보려면 `APP_DEBUG=true`로 임시 설정합니다.
3. 프로덕션에서 **절대로** `APP_DEBUG=true`를 설정하지 마세요.

### 10.2 배포 후 500 오류

**증상**: 배포 전에는 작동했던 애플리케이션이 지금은 500을 반환합니다.

**해결책**:
```bash
# 모든 캐시 지우기 및 재빌드
php artisan optimize:clear
php artisan optimize

# 마이그레이션이 최신인지 확인
php artisan migrate:status

# 파일 권한 확인
chmod -R 775 storage
chmod -R 775 bootstrap/cache

# 새 코드의 PHP 구문 오류 확인
php -l app/Http/Controllers/SomeController.php
```

## 11. 큐 / 백그라운드 작업 문제

### 11.1 알림이 발송되지 않음

**증상**: 사용자가 플랫폼 내 알림을 받지 못합니다.

**해결책**:
1. 큐 워커가 실행 중인지 확인합니다: `ps aux | grep queue:work`
2. 실패한 작업 확인: `php artisan queue:failed`
3. 실패한 작업 상세 검토: `php artisan queue:failed --detail`
4. 개발 환경에서는 `QUEUE_CONNECTION=sync`으로 설정하여 작업을 즉시 실행합니다.

### 11.2 실패한 작업이 누적됨

**증상**: 인수인계 준비 페이지에서 실패한 작업 수가 증가합니다.

**해결책**:
```bash
# 실패한 작업 보기
php artisan queue:failed

# 특정 작업 재시도
php artisan queue:retry {job-id}

# 모든 실패한 작업 재시도
php artisan queue:retry all

# 모든 실패한 작업 초기화 (조사 후)
php artisan queue:flush
```

## 12. 권한 문제

### 12.1 스토리지 쓰기 권한 거부됨

**증상**: 파일 업로드가 권한 오류로 실패하거나 로그를 쓸 수 없습니다.

**해결책**:
```bash
chmod -R 775 storage
chmod -R 775 bootstrap/cache
chown -R www-data:www-data storage bootstrap/cache
```

### 12.2 사용자가 특정 기능에 접근할 수 없음

**증상**: API가 사용자 작업에 대해 403 Forbidden을 반환합니다.

**해결책**:
1. 사용자의 역할을 확인합니다: 해당 작업에 적합한 역할인지 확인합니다.
2. 사용자의 `account_status`를 확인합니다: restricted/suspended 사용자는 특정 작업을 수행할 수 없습니다.
3. `app/Policies/`의 관련 Policy 클래스를 검토합니다.
